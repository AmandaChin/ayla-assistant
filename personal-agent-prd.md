# 个人 Agent 工作台 PRD

版本：V0.1  
日期：2026-05-04  
状态：草案  

## 1. 背景

用户日常工作和学习信息分散在飞书群聊、飞书文档、临时备忘、本地资料、网页资料和个人知识库中。当前主要问题是：

- 群聊信息量大，重要结论、任务、风险容易被淹没。
- 临时备忘缺少统一入口，事后整理成本高。
- 工作资料、学习资料、可公开沉淀资料混杂，缺少清晰的权限和脱敏流程。
- 飞书文档适合协作，但个人长期知识管理、图谱关联和本地归档能力不足。
- Obsidian 适合本地知识库和图谱，但缺少自动采集、自动整理、任务抽取和飞书联动。

因此需要建设一个个人 Agent 工作台，将「信息采集、智能整理、TODO 抽取、知识沉淀、图谱关联、脱敏发布」串成稳定流程。

## 2. 产品定位

本产品是一个本地优先的个人 Agent 工作台，面向用户的日常工作和学习场景，提供电脑端可视化界面，并打通飞书与 Obsidian。

产品不追求让 AI 自动替用户决定一切，而是提供一个可信、可审计、可确认的个人信息处理系统：

- AI 负责采集、总结、分类、抽取、关联和生成候选结果。
- 用户负责确认、修改、归档、发布和授权高风险动作。
- 系统负责保存来源、状态、审计日志和长期知识结构。

## 3. 产品目标

### 3.1 核心目标

1. 提供电脑端 UI，支持分区看板、快捷备忘、TODO 管理、知识库和图谱查看。
2. 通过飞书 MCP 或飞书 OpenAPI 获取指定范围内的群聊消息，并自动生成摘要、备忘和 TODO 候选项。
3. 提供简单可操作的可视化入口，支持随时记录备忘录并自动分区整理。
4. 打通飞书文档和 Obsidian，将工作、学习资料整理为结构化知识库和知识图谱。
5. 将可脱敏的学习资料单独存放到本地发布目录，并支持定期上传到 GitHub。
6. 保证所有敏感动作可控、可回溯，默认经过用户确认。

### 3.2 非目标

V0.1 阶段暂不追求：

- 完全自动读取所有飞书群聊。
- 自动发布任何未经确认的内容到 GitHub。
- 替代飞书、Obsidian 或项目管理工具。
- 构建复杂企业级权限系统。
- 一开始就建设完整图数据库和复杂 GraphRAG 能力。

## 4. 目标用户

### 4.1 主要用户

个人用户本人，具备以下特征：

- 日常高度依赖飞书沟通和飞书文档。
- 经常需要从群聊中提取结论、TODO、风险和备忘。
- 有长期学习、工作沉淀、知识复盘和资料归档需求。
- 希望把部分可脱敏学习资料沉淀到本地，并同步到 GitHub。
- 希望通过 Obsidian 查看本地知识库和图谱关系。

### 4.2 使用场景

- 每天早上查看昨日群聊摘要和今日待办。
- 会议或沟通后快速记录备忘，并自动归入对应项目。
- 从飞书群聊中抽取「我需要做什么」和「哪些信息值得沉淀」。
- 将飞书文档、学习笔记、本地资料整理成 Obsidian 可用的 Markdown 知识库。
- 查看项目、人物、概念、文档、任务之间的知识图谱。
- 定期生成脱敏后的学习资料，并同步到 GitHub。

## 5. 核心体验

### 5.1 一句话体验

打开电脑端工作台，即可看到今天需要关注什么、有哪些待办、哪些群聊值得回看、哪些资料已经沉淀成知识库。

### 5.2 关键原则

- 本地优先：用户的原始数据、配置、知识库和审计记录优先保存在本机。
- 人审优先：写回飞书、归档长期知识库、上传 GitHub 等动作默认需要确认。
- 来源可追溯：每条摘要、TODO、知识笔记都可以回溯到原始消息、文档或备忘。
- 分层整理：先收集，再摘要，再结构化，再图谱化，避免直接把噪声灌进图谱。
- Obsidian 友好：本地知识库以 Markdown 为主，天然支持 Obsidian 链接和图谱。
- 安全发布：可公开资料必须经过脱敏检测和人工确认。

## 6. 信息架构

电脑端 UI 分为以下主区域：

1. 今日看板
2. 收件箱
3. TODO 中心
4. 备忘录
5. 知识库
6. 知识图谱
7. 脱敏发布
8. 设置中心

## 7. 功能需求

## 7.1 今日看板

### 功能说明

今日看板是用户打开应用后的默认首页，用于聚合今日最重要的信息。

### 功能点

- 展示今日待办。
- 展示昨日或最近一次群聊摘要。
- 展示待确认的 AI 候选项。
- 展示最近更新的知识库条目。
- 展示近期风险、阻塞和需要跟进的人或项目。
- 支持按「工作」「学习」「个人」「项目」过滤。

### 用户操作

- 点击 TODO 可进入详情。
- 点击群聊摘要可查看来源消息。
- 点击待确认项可执行确认、编辑、忽略、归档。
- 点击知识条目可跳转到内置知识库或 Obsidian 文件。

### 验收标准

- 用户可以在首页 30 秒内了解今天最重要的事项。
- 每条卡片都能看到来源和更新时间。
- 看板支持手动刷新和自动定时刷新。

## 7.2 飞书群聊采集与总结

### 功能说明

系统通过飞书 MCP 或飞书 OpenAPI 读取用户授权范围内的群聊消息，并生成结构化摘要、备忘和 TODO 候选项。

### 范围控制

V0.1 只支持用户手动配置关注群，不做全量群聊扫描。

每个关注群支持配置：

- 是否启用采集。
- 采集频率。
- 摘要时间窗口。
- 是否只关注提到我的消息。
- 是否抽取 TODO。
- 是否抽取风险和决策。
- 是否允许进入知识库候选。

### 摘要内容

每次群聊总结应包含：

- 核心结论。
- 重要上下文。
- 待办事项。
- 风险和阻塞。
- 关键人物和项目。
- 值得沉淀的知识点。
- 原始消息来源索引。

### TODO 抽取

TODO 候选项字段：

- 标题。
- 描述。
- 来源群聊。
- 来源消息。
- 相关项目。
- 负责人。
- 截止时间。
- 优先级。
- 置信度。
- 状态。

### 用户确认机制

AI 生成的结果默认进入「待确认」状态，用户可以：

- 确认。
- 编辑后确认。
- 忽略。
- 标记为误识别。
- 归入指定项目或知识库分区。

### 验收标准

- 用户可以配置至少 1 个飞书群作为关注群。
- 系统可以按时间窗口生成群聊摘要。
- 摘要中的 TODO 可以进入 TODO 中心。
- 每条摘要和 TODO 都能回溯到原始消息来源。
- 未确认内容不会自动进入长期知识库或发布目录。

## 7.3 快捷备忘录

### 功能说明

用户可以通过桌面端入口或全局快捷键快速记录备忘，系统自动进行分类、标签提取和后续整理。

### 输入方式

- 桌面端输入框。
- 全局快捷键呼出小窗。
- 今日看板快速输入。
- 后续可扩展语音输入、截图输入和浏览器剪藏。

### 备忘字段

- 正文。
- 创建时间。
- 手动分区。
- 自动分类。
- 标签。
- 相关项目。
- 是否 TODO。
- 是否知识库候选。
- 来源类型：手动输入。

### 自动整理

系统应自动判断：

- 是否是 TODO。
- 是否属于某个已有项目。
- 是否与已有笔记重复或相似。
- 是否适合进入知识库。
- 是否需要用户补充信息。

### 验收标准

- 用户可以在 3 秒内完成一条备忘输入。
- 备忘可以自动进入对应分区候选。
- 用户可以在收件箱中批量确认或修改分类。

## 7.4 收件箱

### 功能说明

收件箱承载所有待处理信息，包括飞书摘要、TODO 候选、手动备忘、文档摘录和学习资料。

### 信息状态

- 未处理。
- 待确认。
- 已确认。
- 已归档。
- 已忽略。
- 需补充。
- 已发布。

### 操作能力

- 批量确认。
- 批量忽略。
- 修改分类。
- 关联项目。
- 转为 TODO。
- 转为知识笔记。
- 标记为敏感。
- 加入脱敏候选。

### 验收标准

- 所有 AI 生成内容先进入收件箱。
- 用户可以批量处理候选内容。
- 每条内容都有来源、分类、状态和操作记录。

## 7.5 TODO 中心

### 功能说明

TODO 中心用于管理从群聊、备忘和文档中抽取出来的个人任务。

### 视图

- 今日待办。
- 本周待办。
- 按项目。
- 按来源。
- 按优先级。
- 已完成。

### 任务字段

- 标题。
- 描述。
- 状态。
- 优先级。
- 截止时间。
- 项目。
- 来源。
- 负责人。
- 相关文档。
- 相关知识点。

### 状态流转

- 候选。
- 待办。
- 进行中。
- 已完成。
- 已取消。
- 已归档。

### 验收标准

- 用户可以从群聊摘要和备忘中生成 TODO。
- 用户可以编辑 TODO 的标题、截止时间、项目和优先级。
- TODO 可回溯到来源消息、备忘或文档。

## 7.6 知识库

### 功能说明

知识库用于沉淀工作、学习、项目和方法论内容。底层以本地 Markdown 文件为主，兼容 Obsidian。

### 知识库分区

- 工作记录。
- 项目笔记。
- 学习资料。
- 方法论。
- 会议纪要。
- 人物与协作。
- 待整理资料。
- 可公开资料。

### Markdown 规范

每条知识笔记应包含 Frontmatter：

```yaml
---
title:
type:
tags:
projects:
source:
source_url:
created_at:
updated_at:
sensitivity:
publishable:
---
```

### Obsidian 兼容

系统生成的 Markdown 应支持：

- 双链语法：`[[项目名]]`、`[[概念名]]`。
- 标签语法：`#tag`。
- 文件夹分区。
- YAML Frontmatter。
- 图谱关系查看。

### 验收标准

- 用户可以在桌面端查看知识库列表和详情。
- 知识库文件可以被 Obsidian 正常打开。
- 系统生成的笔记包含 Frontmatter 和来源信息。
- 用户可以从 UI 跳转到对应 Obsidian 文件。

## 7.7 知识图谱

### 功能说明

知识图谱用于展示人、项目、概念、任务、文档、资料之间的关系。

### V0.1 图谱来源

- Obsidian Markdown 双链。
- Markdown Frontmatter。
- 系统抽取的实体关系。
- TODO 和项目之间的关联。
- 飞书群聊摘要中的人物、项目和决策。

### 实体类型

- Person：人物。
- Project：项目。
- Topic：主题。
- Concept：概念。
- Document：文档。
- Task：任务。
- Memo：备忘。
- Resource：资料。

### 关系类型

- belongs_to：属于。
- mentions：提到。
- depends_on：依赖。
- owns：负责。
- related_to：相关。
- derived_from：来源于。
- blocked_by：被阻塞。
- references：引用。

### 图谱视图

- 全局图谱。
- 当前项目图谱。
- 当前笔记局部图谱。
- 人物协作关系。
- TODO 依赖关系。
- 学习主题关系。

### 验收标准

- 用户可以在桌面端查看基础图谱。
- 点击图谱节点可以查看对应笔记、任务或来源。
- 图谱支持按实体类型和分区过滤。
- Obsidian 内也能通过 Markdown 双链查看图谱。

## 7.8 飞书文档同步

### 功能说明

系统支持将确认后的工作记录、会议总结、周报和知识笔记写入飞书文档。

### 同步方向

V0.1 支持：

- 本地知识笔记同步到指定飞书文档。
- 群聊摘要生成飞书文档草稿。
- TODO 和周报内容生成飞书文档草稿。

暂不默认支持：

- 飞书文档全量反向同步。
- 自动覆盖用户编辑过的飞书文档。

### 同步策略

- 用户选择目标文档或模板。
- AI 生成文档草稿。
- 用户确认后写入飞书。
- 系统记录同步日志和文档链接。

### 验收标准

- 用户可以将一条确认后的摘要写入指定飞书文档。
- 写入前展示预览。
- 写入后保存飞书文档链接和同步时间。

## 7.9 脱敏发布与 GitHub 同步

### 功能说明

系统支持将可脱敏的学习资料和知识笔记生成公开版本，存放在本地 `publishable/` 目录，并定期同步到 GitHub。

### 脱敏检测

需要检测并提示：

- 人名。
- 群名。
- 公司内部项目名。
- 内部链接。
- 代码仓库地址。
- 接口地址。
- 账号、Token、Cookie、密钥。
- 未公开业务数据。
- 飞书文档链接。

### 发布流程

```text
资料进入
→ 标记为学习资料或可发布候选
→ AI 生成脱敏版
→ 系统展示脱敏 diff
→ 用户确认
→ 写入 publishable/
→ 本地 Git 提交
→ 定期或手动推送 GitHub
```

### 目录建议

```text
agent-vault/
  private/
    raw_messages/
    work_notes/
    personal_memos/
  obsidian/
    work/
    study/
    projects/
    methods/
    people/
  publishable/
    study_notes/
    sanitized_graph/
  system/
    database.sqlite
    sync_logs/
    prompts/
```

### 验收标准

- 未确认内容不会进入 `publishable/`。
- 每次发布前必须展示脱敏检查结果。
- GitHub 同步只允许同步 `publishable/` 目录。
- 系统保存每次发布记录和对应源文件。

## 8. Agent 工作流

## 8.1 总体流程

```text
采集信息
→ 标准化为统一事件
→ 进入收件箱
→ AI 生成候选摘要 / TODO / 笔记 / 图谱关系
→ 用户确认
→ 写入 TODO / 知识库 / 飞书文档 / Obsidian
→ 可发布内容进入脱敏流程
→ 用户确认后同步 GitHub
```

## 8.2 Agent 角色

### Collector Agent

负责从飞书、手动备忘、本地文件和文档中采集信息。

### Summarizer Agent

负责总结群聊、会议和长文档。

### Task Extractor Agent

负责抽取 TODO、截止时间、负责人和上下文。

### Knowledge Curator Agent

负责将确认后的内容整理为知识库笔记。

### Graph Builder Agent

负责抽取实体关系，生成 Markdown 双链和图谱数据。

### Sanitizer Agent

负责识别敏感信息，生成脱敏版本。

### Review Agent

负责将 AI 候选项组织成用户可确认、可修改的审核界面。

## 10. 权限与安全

### 10.1 安全原则

- 默认最小权限。
- 原始消息和私人备忘只存本地。
- 写回飞书、写入长期知识库、上传 GitHub 均需要用户确认。
- 所有 AI 生成内容必须保留来源。
- 发布内容必须经过脱敏检查。
- 账号密钥、Token 和 Cookie 不得进入知识库和发布目录。

### 10.2 风险动作

以下动作必须二次确认：

- 写入飞书文档。
- 批量归档知识库。
- 批量生成公开资料。
- GitHub push。
- 删除原始数据。
- 修改脱敏规则。

## 11. 非功能需求

### 11.1 性能

- 应用冷启动时间目标小于 3 秒。
- 今日看板刷新目标小于 2 秒。
- 群聊摘要可异步执行，不阻塞 UI。
- 10000 条本地事件以内查询应保持流畅。

### 11.2 可用性

- 备忘录输入路径不超过 2 步。
- 重要候选内容可批量处理。
- 所有自动化任务可暂停、重试和查看日志。

### 11.3 稳定性

- 飞书接口失败时保留重试队列。
- 摘要任务失败时保留原始消息。
- 写入飞书或 GitHub 失败时不得丢失本地数据。

### 11.4 可维护性

- 数据采集、AI 整理、知识库写入、图谱构建、发布同步应模块化。
- Prompt、脱敏规则和同步规则应可配置。
- 保留审计日志，便于追踪错误来源。

## 15. 主要风险

### 15.1 飞书权限风险

风险：飞书消息读取受 Bot 权限、群权限和接口限制影响。  
应对：V0.1 只支持用户手动配置关注群，并清晰展示采集状态和失败原因。

### 15.2 隐私和误发布风险

风险：敏感内容可能被错误归档或发布。  
应对：发布默认人工确认，GitHub 只同步 `publishable/`，并提供脱敏 diff。

### 15.3 AI 误判风险

风险：AI 可能错误抽取 TODO、误分类或制造不准确关系。  
应对：所有 AI 结果先进入待确认状态，保留置信度和来源。

### 15.4 图谱失控风险

风险：过早构建复杂图谱会产生大量低质量节点。  
应对：先用 Obsidian 双链和轻量实体关系表，等知识库稳定后再扩展图数据库。

### 15.5 工具链复杂度风险

风险：飞书、Obsidian、GitHub、本地 UI、AI 编排同时建设，复杂度过高。  
应对：按阶段推进，先做本地工作台和 Obsidian，再打通飞书和发布流程。
